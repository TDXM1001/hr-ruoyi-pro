# 2026-03-22 不动产占用管理前端闭环设计

## 1. 背景

截至 2026-03-22，不动产占用管理后端最小闭环已经完成：

1. 占用单表已落库
2. 占用查询 / 发起 / 变更 / 释放接口已具备
3. 生命周期已可返回 `occupancyRecords`

但前端“占用”页签仍停留在只读展示，实际没有使用占用单数据，也没有发起占用、变更占用、释放占用的操作入口。

## 2. 本轮目标

本轮只做详情页内的占用前端闭环，不新增独立页面。

要完成：

1. 占用 Tab 展示“当前有效占用卡片”
2. 占用 Tab 展示“占用历史记录列表”
3. 无有效占用时支持“发起占用”
4. 有有效占用时支持“变更占用”“释放占用”
5. 使用 Drawer 承载表单，动作完成后刷新详情壳数据
6. 补一条本地点测样例 SQL，保证页面可立即验证

本轮不做：

1. 占用审批
2. 占用附件
3. 占用批量操作
4. 空间粒度建模

## 3. 方案对比

### 3.1 方案 A：占用页签内嵌表单

优点：切换少。

缺点：

1. 当前占用、历史记录、表单会挤在一个阅读页里
2. 表单状态和详情状态混杂，后续维护成本高

### 3.2 方案 B：占用页签 + Drawer 表单（推荐）

优点：

1. 保持详情页阅读感
2. 操作不离开资产上下文
3. 字段量适中，Drawer 承载刚好
4. 返工最少，符合现有详情壳结构

缺点：

1. 需要管理 Drawer 状态与回刷逻辑

### 3.3 方案 C：占用独立页面

优点：边界最清晰。

缺点：

1. 对当前详情壳来说过重
2. 用户在详情和占用之间来回跳页，体验割裂

### 3.4 结论

采用方案 B：`占用页签 + Drawer 表单`。

## 4. 页面结构

占用页签拆成 3 个区块：

1. 顶部提示条
- 明确占用动作只影响当前资产的在用归属，不处理审批

2. 当前占用卡片
- 资产编码
- 当前状态
- 使用部门
- 责任人
- 使用位置
- 占用开始日期
- 最近一次变更说明
- 操作区：`发起占用 / 变更占用 / 释放占用`

3. 占用历史列表
- 占用单号
- 状态标签
- 使用部门 / 责任人 / 位置
- 起止日期
- 发起/变更原因
- 释放原因

## 5. 交互规则

### 5.1 发起占用

显示条件：当前无 `ACTIVE` 占用。

Drawer 字段：

1. 使用部门
2. 责任人
3. 使用位置
4. 占用开始日期
5. 发起原因
6. 备注

提交成功后：

1. 关闭 Drawer
2. 刷新详情与生命周期
3. 当前占用卡片切为最新占用
4. 历史列表新增一条 `ACTIVE`

### 5.2 变更占用

显示条件：当前存在 `ACTIVE` 占用。

Drawer 默认带出当前有效占用：

1. 使用部门
2. 责任人
3. 使用位置
3. 新占用开始日期
4. 变更原因
5. 备注

提交成功后：

1. 原占用进入历史，状态变 `RELEASED`
2. 新占用成为当前占用
3. 刷新详情与生命周期

### 5.3 释放占用

显示条件：当前存在 `ACTIVE` 占用。

Drawer 字段：

1. 释放日期
2. 释放原因
3. 备注

提交成功后：

1. 当前占用卡片切为“暂无有效占用”
2. 历史列表中当前单状态变为 `RELEASED`
3. 详情页的使用部门/责任人/位置同步变空

## 6. 数据来源

### 6.1 读取

优先使用生命周期接口返回的 `occupancyRecords`，不再继续用 `handoverRecords` 冒充占用历史。

仍保留 `handoverRecords`，但它回到“交接轨迹”的辅助信息角色，不再作为占用主数据。

### 6.2 写入

新增前端 API：

1. `listRealEstateOccupancies`
2. `addRealEstateOccupancy`
3. `changeRealEstateOccupancy`
4. `releaseRealEstateOccupancy`

提交成功后统一调用详情页 `reloadDetail()`，重新拉取：

1. `getRealEstateDetail`
2. `getRealEstateLifecycle`
