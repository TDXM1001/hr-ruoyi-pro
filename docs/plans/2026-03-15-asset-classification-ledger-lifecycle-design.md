# 资产分类、台账与生命周期分层设计

## 背景

现有资产模块已经完成了资产聚合模型的前后端第一轮对接，资产台账、聚合详情、动态属性值、附件、财务弹窗和领用流程的基础链路已经打通，但当前仍存在一个核心认知混淆：

- 资产分类页面可以新增分类类目，但前端没有“分类属性模板管理”入口
- 资产台账编辑抽屉中的“扩展信息”页签只能消费分类模板并填写实例值，不能新增模板本身
- 固定资产与不动产当前仍共用一套台账入口，但后续生命周期动作并不应继续混用

结合现有文档与代码状态，本次设计的目标不是继续堆叠工作流入口，而是先把“分类模板、台账实例、生命周期动作”三层职责拆清楚，为后续固定资产和不动产分叉演进建立稳定边界。

## 设计目标

- 将“分类模板定义”和“台账实例取值”彻底解耦
- 明确固定资产与不动产共享什么、不共享什么
- 继续以 `assetId` 作为内部主键，`assetNo` 作为业务编码
- 把工作流从当前主线中后置，避免在主数据边界未稳定时过早耦合审批
- 为后续“编辑、变更、删除、处置”等基础动作建立统一口径

## 设计结论

本次设计采用三层分离模型：

1. 分类层：只负责模板定义
2. 台账层：只负责资产实例维护
3. 生命周期层：只负责实例动作与审批承载

三层关系如下：

```text
资产分类层（模板定义）
├── 分类树
├── 分类编码
└── 动态属性模板

资产台账层（实例维护）
├── 基础信息
├── 财务信息
├── 不动产信息
├── 动态属性值
└── 附件

生命周期层（实例动作）
├── 固定资产动作：领用/归还/维修/报废/处置
├── 不动产动作：权属变更/用途变更/状态变更/注销或处置
└── 工作流：仅挂在关键动作上
```

## 一、分层边界

### 1. 分类层

分类层只负责“定义字段长什么样”，不承载任何具体资产实例值。

分类层职责：

- 维护分类树、分类编码和层级关系
- 维护分类下的动态属性模板
- 定义字段编码、字段名称、数据类型、默认值、是否必填、选项来源和校验规则
- 对模板字段做保留字段校验和分类内唯一性校验

分类层不负责：

- 维护某一条资产的扩展属性值
- 承担固定资产或不动产的业务动作
- 直接挂接审批流程

### 2. 台账层

台账层只负责“某个资产实例的聚合主数据”，不负责定义模板。

台账层职责：

- 以 `assetId` 为内部主键维护单个资产
- 使用统一的五段式聚合结构：
  - `basicInfo`
  - `financeInfo`
  - `realEstateInfo`
  - `dynamicAttrs`
  - `attachments`
- 在编辑时根据 `categoryId` 拉取模板并填写对应实例值
- 保持固定资产与不动产共用一套聚合详情与保存接口

台账层不负责：

- 新增或修改分类模板定义
- 用领用、报废、维修等生命周期动作去污染主数据结构
- 把工作流语义混进普通编辑保存

### 3. 生命周期层

生命周期层只负责“实例动作”，并且固定资产与不动产在这一层开始分叉。

固定资产优先支持：

- 领用
- 归还
- 维修
- 报废
- 处置

不动产优先支持：

- 权属变更
- 用途变更
- 状态变更
- 注销或处置

生命周期层的核心原则：

- 所有动作统一依赖资产主上下文
- 不动产不复用固定资产的“领用、维修、报废”语义
- 工作流仅服务于关键生命周期动作，不服务于分类模板维护和普通台账编辑

## 二、页面与前后端职责

### 1. 资产分类页

当前 [index.vue](/e:/my-project/hr-ruoyi-pro/art-design-pro/src/views/asset/category/index.vue) 已支持分类树展示与分类弹窗维护，但缺少属性模板管理区域。后续推荐在分类页中新增“属性模板管理”能力，而不是把模板维护塞回资产台账抽屉。

推荐形态：

- 左侧或上方保留分类树/分类表格
- 右侧抽屉、下方卡片或页内分栏展示当前分类的属性模板列表
- 模板新增、编辑、禁用、删除统一在分类页完成

前端职责：

- 展示当前分类下的属性模板列表
- 提供新增、编辑、禁用、删除入口
- 对保留字段编码和基础必填项做前端提示

后端职责：

- 提供模板 CRUD 与按分类查询接口
- 校验保留字段编码
- 校验同一分类下 `attrCode` 唯一

### 2. 资产台账页

当前 [asset-edit-drawer.vue](/e:/my-project/hr-ruoyi-pro/art-design-pro/src/views/asset/list/modules/asset-edit-drawer.vue) 中的“扩展信息”页签应继续保留，但角色必须明确为“实例填值”。

前端职责：

- 选择 `categoryId` 后拉取该分类的模板定义
- 按模板渲染文本、数字、日期、JSON、选项型控件
- 回填已有资产的扩展属性值
- 提交时将表单映射为 `dynamicAttrs[]`

后端职责：

- 返回聚合详情中的模板值对应数据
- 保存实例级动态属性值
- 在模板被禁用、字段缺失或主键不匹配时返回明确错误

明确禁止：

- 在台账页里直接新增模板字段
- 在台账页里把模板定义和实例取值混在一套对象里

### 3. 生命周期页

生命周期页在本设计中先只做边界定义和入口约束，不把工作流作为当前阶段的主实现目标。

前端职责：

- 从台账页统一带出动作上下文：
  - `assetId`
  - `assetNo`
  - `assetName`
  - `assetType`
  - `assetStatus`
- 基于 `assetType` 控制哪些动作对当前资产可见
- 对未落地的动作入口给出明确提示，不保留必然失败的按钮

后端职责：

- 基于 `assetId` 与 `assetType` 校验动作是否合法
- 严格区分固定资产动作和不动产动作
- 对关键动作预留工作流接入点

## 三、数据与接口口径

### 1. 分类模板接口口径

分类接口和分类属性接口必须分工明确：

- `GET /asset/category/list`：分类树或分类列表
- `GET /asset/category/{id}`：分类详情
- `GET /asset/categoryAttr/category/{categoryId}`：按分类查询模板
- `POST /asset/categoryAttr`：新增模板
- `PUT /asset/categoryAttr`：修改模板
- `PUT /asset/categoryAttr/disable/{attrId}`：禁用模板
- `DELETE /asset/categoryAttr/{attrIds}`：删除模板

约束：

- 分类页维护模板
- 台账页只允许读取模板，不允许修改模板

### 2. 台账聚合结构口径

台账实例统一保持以下五段式聚合结构：

```ts
{
  basicInfo,
  financeInfo,
  realEstateInfo,
  dynamicAttrs,
  attachments
}
```

其中：

- `basicInfo.assetId` 是内部主键
- `basicInfo.assetNo` 是业务编码
- `dynamicAttrs` 只承载实例值
- `realEstateInfo` 仅在不动产场景下强制存在

### 3. 动态属性模板与实例值口径

模板定义和实例值必须在接口层完全分离。

模板定义最小字段：

- `attrId`
- `categoryId`
- `attrCode`
- `attrName`
- `dataType`
- `isRequired`
- `defaultValue`
- `optionSourceType`
- `optionSource`
- `validationRule`
- `status`

实例值最小字段：

- `assetId`
- `attrId`
- `attrCode`
- `attrValueText`
- `attrValueNumber`
- `attrValueDate`
- `attrValueJson`

结论：

- 分类页维护“字段长什么样”
- 台账页维护“该资产字段填了什么值”

### 4. 生命周期动作口径

所有生命周期动作统一携带资产主上下文：

- `assetId`
- `assetNo`
- `assetName`
- `assetType`
- `assetStatus`

在此基础上再按动作附加业务字段。固定资产与不动产可以共用“资产引用模型”，但不能共用同一套动作语义。

### 5. 保留字段与错误口径

前后端双侧都必须保留以下校验：

- `asset_no`
- `asset_name`
- `original_value`
- `property_cert_no`

同时明确：

- 同一分类下 `attrCode` 唯一
- 模板缺失或模板已禁用时，不应默默吞掉实例值
- `assetType` 不匹配的生命周期动作必须由后端拒绝执行
- 已进入关键状态的资产，“编辑”和“变更”不能混用

## 四、实施路线

### 第一阶段：补齐分类模板管理

目标是先解决“动态属性无法新增”的真实问题。

范围：

- 前端分类页补齐属性模板管理入口
- 对接后端模板 CRUD
- 明确保留字段提示、启用状态和模板列表展示

阶段完成标志：

- 分类页可新增、编辑、禁用、删除模板
- 台账页选择分类后能消费最新模板

### 第二阶段：收口台账主数据

目标是把资产实例维护稳定下来，不把生命周期和审批提前卷入。

范围：

- 固定资产新增、编辑、变更、删除
- 不动产新增、编辑、变更、删除
- 动态属性值填写与回填
- 附件、财务、不动产信息聚合保存

阶段完成标志：

- 台账页扩展信息只负责实例值
- 固定资产和不动产共用统一聚合台账入口

### 第三阶段：按资产类型拆分生命周期

目标是让固定资产和不动产在“动作层”开始分叉。

范围：

- 固定资产动作：领用、归还、维修、报废、处置
- 不动产动作：权属变更、用途变更、状态变更、注销或处置

阶段完成标志：

- 前端动作入口受 `assetType` 控制
- 后端动作校验不再混用固定资产与不动产语义

### 第四阶段：最后接工作流

目标是仅把审批挂到关键动作上。

范围：

- 分类模板维护不进工作流
- 普通台账编辑默认不进工作流
- 报废、处置、权属变更等关键动作再接审批

阶段完成标志：

- 工作流只服务于关键生命周期动作
- 页面中不再出现“看似可点、实际上必失败”的审批入口

## 五、编辑、变更、删除的语义约束

为了避免后续业务口径混乱，本设计建议把三个常用动作先定义清楚：

- 编辑：修正主数据录入错误，不强调业务留痕
- 变更：对已生效资产发起正式调整，应留痕，必要时进入审批
- 删除：仅适用于误建、草稿或允许清理的资产，不等同于报废或注销

结论：

- 固定资产和不动产都应具备“新增、编辑、变更、删除”基础能力
- 领用、维修、报废等属于固定资产专属生命周期动作
- 不动产应拥有自己的生命周期动作集合，不强行复用固定资产业务名词

## 六、测试与验收标准

### 1. 分类模板管理验收

- 资产分类页可以维护动态属性模板
- 保留字段编码会被前后端同时拦截
- 同一分类下模板编码不能重复

### 2. 台账实例验收

- 资产台账“扩展信息”只消费模板，不负责新增模板
- 切换分类后实例字段按模板正确变化
- 旧资产编辑时动态属性值可以正确回填

### 3. 生命周期边界验收

- 固定资产和不动产的动作入口按 `assetType` 分流
- 不动产不会展示错误的固定资产业务动作
- 未落地接口不保留失败按钮

### 4. 工作流后置验收

- 本阶段设计中不把工作流接入分类模板维护
- 不把工作流接入普通台账编辑
- 关键动作才作为后续审批接入点

## 七、非目标

以下内容不属于本轮主设计的直接交付范围：

- 立即补齐全部工作流接口
- 在台账页中直接维护分类属性模板
- 让不动产复用固定资产的领用、维修、报废页面
- 用动态属性模板去替代财务或不动产核心主字段

## 结论

后续资产模块应围绕“分类模板、台账实例、生命周期动作”三层推进。分类页负责定义模板，台账页负责维护实例，生命周期页负责承载动作与审批接入点。固定资产与不动产共享统一资产主键和聚合台账，但从生命周期开始分叉，工作流作为关键动作的后置能力接入。
