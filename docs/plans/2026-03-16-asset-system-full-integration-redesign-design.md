# 资产系统全菜单联调重构设计

## 1. 背景

当前仓库中的资产系统已经完成过一轮前后端聚合模型改造，前端位于 `art-design-pro`，后端位于 `RuoYi-Vue`。现状并不是“完全没有资产能力”，而是出现了以下典型问题：

- 左侧资产系统菜单已经展开到全链路：资产分类、资产台账、领用归还、维修管理、报废处置、权属变更、用途变更、状态变更、注销处置。
- 前后端已经引入 `assetId`、聚合详情、动态属性、附件、财务重算、报表预警等能力，但并未在全部菜单中完全收口。
- 部分历史文档已经定义了新的契约口径，部分代码已经切到新口径，仍存在“文档已更新、页面未完全接入”或“页面已改造、业务动作仍沿用旧模型”的风险。
- 固定资产业务和不动产业务都依附于资产主档，但当前联调容易被拆成若干孤立页面，最终导致每个菜单都能运行，系统整体仍然不统一。

本次工作不是单纯修一个页面，而是将左侧整个资产系统定义为一个“统一契约重构 + 分阶段联调落地项目”。

## 2. 目标与非目标

### 2.1 目标

- 覆盖左侧资产系统全部菜单，形成统一的前后端联调口径。
- 明确资产主档、分类模板、业务动作单据、流程任务之间的边界。
- 统一主键、状态枚举、请求体、响应体、错误回显和权限标识。
- 形成可拆分、可并行、可验收的实施方案文档。
- 保证固定资产和不动产都运行在同一套资产主档模型之下。

### 2.2 非目标

- 本文档不直接进行代码实现。
- 本文档不重新设计审批引擎本身，只收敛资产系统与审批中心的对接方式。
- 本文档不扩展新的业务菜单，仅收口现有左侧菜单能力。

## 3. 方案选择

### 方案 A：按菜单逐页联调

- 以左侧菜单为单位推进，一个菜单一个菜单修复。
- 优点：边界清晰，方便排期。
- 缺点：公共契约容易重复修改，状态口径和资产引用模型难以统一。

### 方案 B：按领域契约分层联调

- 先统一资产主档、分类、财务、流程、报表等公共契约，再按菜单落地。
- 优点：能一次性收敛 `assetId`、聚合请求体、统一动作单据模型和状态口径。
- 缺点：前期设计和梳理工作更多。

### 方案 C：按业务链路端到端联调

- 优先打通分类、台账、固定资产业务链路，再补不动产动作页。
- 优点：容易较早形成业务闭环。
- 缺点：不动产和治理能力容易被滞后，最后仍可能形成两套模型。

### 推荐

推荐采用方案 B。

本次目标是“全菜单完美对接”，重点是统一契约与模块边界，而不是继续围绕单个页面打补丁。现有仓库已经具备一部分聚合化改造基础，适合围绕公共契约继续收口。

## 4. 总体架构

建议将资产系统拆成 6 个稳定层次：

1. 基础主数据层
2. 固定资产业务动作层
3. 不动产业务动作层
4. 流程协同层
5. 分析治理层
6. 公共支撑层

### 4.1 基础主数据层

- 资产分类
- 资产台账
- 动态属性
- 附件
- 财务快照

### 4.2 固定资产业务动作层

- 领用归还
- 维修管理
- 报废处置

### 4.3 不动产业务动作层

- 权属变更
- 用途变更
- 状态变更
- 注销处置

### 4.4 流程协同层

- 待办
- 已办
- 审批动作回写资产状态与业务单据状态

### 4.5 分析治理层

- 报表汇总
- 预警
- 归档与历史查询

### 4.6 公共支撑层

- 字典
- 部门/人员选择
- 权限标识
- 上传组件
- 统一状态枚举
- 统一错误码和回显格式

## 5. 核心领域模型

### 5.1 资产主档聚合 `AssetAggregate`

资产主档是整个系统唯一事实源，由以下部分组成：

- `basicInfo`
- `financeInfo`
- `realEstateInfo`
- `dynamicAttrs`
- `attachments`

设计要求：

- 资产台账是资产主档唯一维护入口。
- 其他业务动作菜单只能引用资产主档，不得维护可编辑的资产事实字段。
- 不动产不是一套独立主档，而是 `AssetAggregate` 下 `realEstateInfo` 更丰富的资产类型分支。

### 5.2 分类模板 `AssetCategoryTemplate`

分类模块不仅是分类树，还决定资产表单结构与动态属性规则：

- 分类树
- 分类属性定义
- 默认值
- 校验规则
- 列表展示与查询配置

设计要求：

- 分类是资产建模入口。
- 动态属性必须由分类统一输出，不允许各业务页重复定义表单扩展字段。

### 5.3 业务动作单据 `AssetBusinessOrder`

所有动作菜单统一抽象为业务单据，最小公共字段包括：

- `bizNo`
- `bizType`
- `assetId`
- `assetNo`
- `assetName`
- `status`
- `wfStatus`
- `reason`
- `extPayload`

设计要求：

- 固定资产业务和不动产业务都必须基于同一套单据模型。
- 菜单差异体现在扩展字段，而不是重建一套主档结构。

### 5.4 流程任务 `WorkflowTask`

待办和已办只关注流程与业务映射：

- 当前单据类型
- 当前节点
- 当前审批状态
- 审批完成后的回写规则

设计要求：

- 流程中心不再二次维护资产事实数据。
- 业务菜单与流程菜单的跳转依据统一为 `bizType + bizNo + wfStatus`。

## 6. 全菜单目标边界

### 6.1 资产分类

前端职责：

- 分类树维护
- 分类属性定义维护
- 校验规则配置
- 启停用与排序管理

后端职责：

- 分类树与属性元数据持久化
- 保留字段冲突校验
- 删除或停用影响校验

### 6.2 资产台账

前端职责：

- 列表查询
- 聚合详情展示
- 新增、编辑、归档
- 财务快照展示
- 动态属性与附件维护
- 跳转各业务动作入口

后端职责：

- 聚合读写
- 财务计算与折旧约束
- 动态属性和附件落库
- 时间线与归档控制

### 6.3 领用归还

前端职责：

- 发起领用申请
- 查看领用台账
- 对可归还单据执行归还

后端职责：

- 校验资产是否可领用
- 处理审批流转
- 审批和归还时回写资产状态

### 6.4 维修管理

前端职责：

- 发起维修申请
- 查看维修单详情
- 执行完工动作

后端职责：

- 校验维修前置条件
- 处理审批与完工流转
- 写入资产时间线与维修履历

### 6.5 报废处置

前端职责：

- 发起报废/处置申请
- 查看处置记录
- 区分报废与处置展示

后端职责：

- 处理审批流转
- 回写资产终态
- 归档处置结果

### 6.6 权属变更

前端职责：

- 发起权属变更
- 展示变更前后权属字段

后端职责：

- 处理审批流转
- 审批通过后回写不动产权属字段

### 6.7 用途变更

前端职责：

- 发起用途变更
- 展示变更前后用途字段

后端职责：

- 执行轻流程或免审批动作
- 回写土地/房屋用途字段

### 6.8 状态变更

前端职责：

- 发起状态变更
- 展示变更前后资产状态

后端职责：

- 校验目标状态合法性
- 回写 `assetStatus`
- 写入状态变更记录

### 6.9 注销处置

前端职责：

- 发起注销/处置申请
- 展示目标状态与处理类型

后端职责：

- 处理审批流转
- 审批通过后将不动产转入终态

## 7. 统一接口契约

### 7.1 统一资产引用模型

所有资产相关页面统一使用以下最小引用对象：

```ts
type AssetRef = {
  assetId: number
  assetNo: string
  assetName: string
  assetType: string
  assetStatus: string
}
```

规则：

- `assetId` 是唯一内部主键。
- `assetNo` 仅用于展示、导出和业务编码。
- 所有业务申请创建时必须传 `assetId`，可兼容携带 `assetNo` 作为快照冗余。

### 7.2 统一资产聚合契约

新增、修改、详情统一采用聚合模型：

```ts
type AssetAggregateReq = {
  basicInfo: AssetBasicInfo
  financeInfo: AssetFinanceInfo
  realEstateInfo: AssetRealEstateInfo | null
  dynamicAttrs: AssetDynamicAttrValue[]
  attachments: AssetAttachment[]
}
```

设计要求：

- 资产台账不再提交平铺单表字段。
- 动态属性与附件必须成为主档标准组成部分。
- 财务只读字段以后端计算结果为准。

### 7.3 统一业务动作单据契约

所有动作菜单统一采用“资产引用 + 单据公共字段 + 扩展负载”的形式。

公共字段包括：

- `bizNo`
- `bizType`
- `assetId`
- `assetNo`
- `assetName`
- `status`
- `wfStatus`
- `reason`

扩展字段按动作菜单不同而变化，但不得破坏公共模型。

### 7.4 统一接口风格

- 列表接口统一分页结构。
- 详情接口统一返回当前页面渲染所需最小完整信息。
- 创建接口统一返回业务单据号或主键。
- 修改接口统一校验当前状态是否合法。
- 字段错误、业务规则错误、系统异常统一采用可区分的回显方式。

## 8. 统一状态体系

必须显式拆分 4 套状态：

1. `assetStatus`
2. `status`
3. `wfStatus`
4. `archiveStatus`

### 8.1 资产状态 `assetStatus`

表示资产本体当前状态，例如：

- 在用
- 领用中
- 维修中
- 待处置
- 已报废
- 已处置
- 已归档

### 8.2 业务单据状态 `status`

表示业务单据自身所处阶段，例如：

- 草稿
- 待审批
- 已通过
- 已驳回
- 已完成
- 已取消

### 8.3 流程状态 `wfStatus`

表示审批流程引擎状态，例如：

- 未启动
- 审批中
- 已完成
- 已驳回

### 8.4 归档状态 `archiveStatus`

表示资产是否仍属于日常台账，例如：

- active
- archived

强制规则：

- 资产台账列表只展示 `assetStatus`。
- 业务单据列表优先展示 `status`，需要时补充 `wfStatus`。
- 流程中心明确区分单据状态和审批状态。
- 报表与预警取数必须按状态维度分别聚合。

## 9. 分阶段实施方案

### 阶段 0：基线确认与冻结

- 输出现状差异矩阵
- 冻结枚举、权限、菜单命名

### 阶段 1：公共契约层重构

- 收敛前端 `types` 与 `api`
- 收敛后端 DTO / VO / Controller 契约
- 收敛统一状态体系与错误回显

### 阶段 2：主数据包落地

- 完成资产分类和资产台账
- 接通动态属性、附件、财务快照
- 让台账成为唯一事实源页面

### 阶段 3：固定资产业务包落地

- 领用归还
- 维修管理
- 报废处置

### 阶段 4：不动产业务包落地

- 权属变更

- 状态变更

### 阶段 5：流程与治理收口

- 待办、已办统一跳转

### 阶段 6：联调回归与上线切换

## 10. 任务拆包建议

建议按 5 个交付包拆分，而不是按文件拆分：

1. 公共契约包
2. 主数据包
3. 固定资产业务包
4. 不动产业务包
5. 流程治理包

多人并行时建议：

- 1 人负责公共契约和后端基础层
- 1 人负责资产分类和资产台账
- 1 人负责固定资产业务
- 1 人负责不动产业务
- 1 人负责流程、报表和联调回归

## 11. 风险

### 11.1 状态口径继续漂移

- 表现：不同页面继续混用 `status` 和 `assetStatus`
- 对策：在公共契约阶段冻结并统一所有状态枚举

### 11.2 资产台账不是唯一事实源

- 表现：业务菜单继续保存可编辑主档字段
- 对策：动作单据只保留资产快照，不再保留可编辑主档字段

### 11.3 固定资产与不动产分裂成两套模型

- 表现：不动产页面单独维护资产主字段
- 对策：所有不动产业务仍挂在统一 `AssetAggregate` 之下

### 11.4 审批回写不一致

- 表现：单据状态变化与资产状态变化不同步
- 对策：把审批回写统一沉到后端事务规则中

### 11.5 权限与菜单不匹配

- 表现：页面能开但按钮无权限，或接口可调但菜单不可见
- 对策：将权限收口单独作为治理阶段工作

### 11.6 历史数据兼容失败

- 表现：旧数据缺少 `assetId` 或状态值不合法
- 对策：准备补数 SQL 和兼容修复脚本，不依赖前端兜底

## 12. 验收标准

### 12.1 基础验收

- 左侧全部资产菜单可正常访问
- 查询、详情、创建、审批、回写能力完整
- 无页面继续把 `assetNo` 作为内部唯一主键

### 12.2 契约验收

- 资产主档统一使用聚合模型
- 所有业务单据统一带 `assetId`
- 状态口径不再混用
- 动态属性、附件、财务能力接入统一主档模型

### 12.3 流程验收

- 待办、已办能准确跳转业务单据
- 审批通过/驳回后，业务状态与资产状态同步正确
- 时间线、台账、报表取数一致

### 12.4 治理验收

- 报表与预警口径和资产主档一致
- 归档资产不污染日常台账
- 菜单、按钮、接口鉴权一致

### 12.5 工程验收

- 前端 lint / build 通过
- 后端核心测试通过
- SQL 迁移可重复执行
- 联调问题闭环后再进入上线切换

## 13. 结论

本次工作应定义为“资产系统全菜单统一契约重构 + 分阶段联调落地项目”，而不是零散的前端补接口工作。只有先收敛主档、状态、单据、流程和治理能力，再按交付包推进，才能实现左侧整个资产系统的完整对接与长期可维护。

## 14. 实施冻结项

- 主键统一为 `assetId`，`assetNo` 仅保留展示与业务编码职责。
- 状态统一拆分为 `assetStatus`、`status`、`wfStatus`、`archiveStatus` 四套口径。
- 左侧菜单名称、层级和权限前缀在本轮执行期内冻结，不再新增同义变体。
- 执行顺序冻结为“基线矩阵 -> 前端共享契约 -> 后端基础契约 -> 菜单接入 -> 流程治理 -> 全量回归”，未经上一批验证通过不跨批次推进。
- 本轮在当前开发分支按批次推进，每批完成后必须回写实施计划状态并保留验证结果。
