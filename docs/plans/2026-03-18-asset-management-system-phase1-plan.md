# Asset Management System Phase 1 Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 在当前仓库中以尽量小的体量交付资产管理系统第一期可试运行版本，先建立统一资产台账底座，并打通固定资产最小闭环。

**Architecture:** 第一期开“统一台账 + 固定资产闭环”的纵向切片，而不是同时铺开固定资产与不动产两条完整业务线。后端建议新增独立 `ruoyi-asset` 业务模块承载资产域，前端在 `art-design-pro` 增加资产菜单与页面，并复用现有用户、部门、角色、字典、导出和权限体系；不动产只保留模型兼容位和后续扩展空间，不在本批次上线业务流程。

**Tech Stack:** Spring Boot 3、MyBatis、MySQL、RuoYi 权限/字典/Excel 导出、Vue 3、TypeScript、Element Plus、art-design-pro

---

## 1. 先把“一期”重新定义

当前设计文档里的“一期 MVP”仍然偏大，已经接近“完整资产系统一期”，不适合作为当前仓库的第一个交付目标。

建议把本项目的第一期重新定义为：

- 统一资产台账底座
- 固定资产正式建账
- 固定资产领用、调拨、退还
- 固定资产盘点
- 固定资产处置
- 财务基础字段维护与导出
- 基础统计与少量预警

明确不放入第一期上线范围：

- 不动产业务流程上线
- 租赁、合同、到期管理
- 审批流引擎
- 移动端盘点
- 外部系统接口集成
- 深度维修工单体系
- 复杂空间粒度管理

## 2. 为什么这样切最合适

- 设计文档中的未决问题大多集中在不动产和审批链上，继续把它们放在第一期会直接抬高范围不确定性。
- 当前仓库还没有任何资产域模块，后端、前端、菜单、权限、SQL、字典、导出都需要从 0 补齐，先做一条完整闭环比先做一堆平台壳子更容易验收。
- 固定资产流程相对标准，最适合验证统一状态、统一台账、交接留痕、盘点闭环、处置回写这几条核心设计。
- 一旦固定资产闭环跑通，第二批再把不动产接到同一套台账和状态体系中，返工会明显少于双线同时推进。

## 3. 第一期开工前的默认业务假设

如果业务方没有新的明确结论，第一期按以下默认假设推进：

- 第一期开通的资产类型只有固定资产，不动产仅保留分类和字段兼容设计。
- 第一期交接支持“单据多资产”批量操作，但单据明细仅允许固定资产，不支持混合不动产。
- 资产来源只考虑手工新增和存量补录，不对接采购系统。
- 处置先做业务登记和结果确认，不做完整审批流。
- 财务协同先做字段维护、导出和对账口径，不做系统接口。
- 盘点先做 PC 端任务和结果回写，不做扫码、RFID、移动端。
- 附件、图片、证照先不作为第一期核心验收项。

## 4. 第一阶段目标不是“大而全”，而是“小而闭环”

第一期验收标准建议收敛为：

- 资产管理员可以维护固定资产台账主档。
- 固定资产可以从建账进入在册、使用中、盘点中、待处置、已处置等主状态。
- 领用、调拨、退还都有交接记录，不允许直接改字段跳过过程。
- 盘点结果能回写台账，差异能进入处置或修正。
- 财务可以拿到原值、取得日期、启用日期、状态等基础导出结果。
- 管理层至少能看到总量、状态分布、闲置和待盘点情况。

## 5. 第一期按四个批次推进

### 批次 0：范围冻结与数据口径冻结

**目标**

把“第一期只做固定资产闭环”这件事书面冻结，避免边做边把不动产、合同、审批流重新带回来。

**完成项**

- 确认第一期上线范围与明确排除项
- 确认固定资产主档最小字段集
- 确认统一状态集在第一期的可用子集
- 确认财务最小导出字段
- 确认固定资产分类层级、编码规则、位置口径
- 确认领用、调拨、退还、盘点、处置的最小单据字段

**输出物**

- 第一期开通范围清单
- 数据字典草案
- 状态流转表
- 财务导出口径表

**验收标准**

- 所有人对“第一期不做什么”达成一致
- 建账、领用、盘点、处置四条路径都有明确入口字段和结果字段

### 批次 1：统一台账底座与主数据

**目标**

先让系统拥有正式的“资产主档”，并把后续流程都挂在主档上，而不是先做零散业务单据。

**建议实现范围**

- 建立资产分类、后端统一生成的编码规则、状态字典
- 建立固定资产主档表
- 建立资产变更记录表
- 建立资产位置、归属、责任字段
- 完成台账列表、详情、新增、编辑、导入或补录
- 完成资产菜单、权限点、基础查询

**建议后端落点**

- 修改 `RuoYi-Vue/pom.xml`
- 修改 `RuoYi-Vue/ruoyi-admin/pom.xml`
- 新增 `RuoYi-Vue/ruoyi-asset/`
- 新增 `RuoYi-Vue/ruoyi-admin/src/main/java/com/ruoyi/web/controller/asset/`
- 新增 `RuoYi-Vue/ruoyi-admin/src/main/resources/mapper/asset/`
- 新增 `RuoYi-Vue/sql/asset/`

**建议前端落点**

- 新增 `art-design-pro/src/api/asset/`
- 新增 `art-design-pro/src/views/asset/ledger/`
- 新增 `art-design-pro/src/views/asset/ledger/form/`
- 新增 `art-design-pro/src/views/asset/ledger/detail/`
- 维护 `RuoYi-Vue/sql/asset/01-asset-seed.sql` 中的动态路由菜单

**验收标准**

- 可以新增固定资产并形成正式台账记录
- 手工新增默认由后端生成资产编号，存量补录允许手工维护历史编号
- 通用编辑页不修改资产编号，编号调整应独立成专门流程
- 台账记录具备分类、编码、归属、责任、位置、原值、取得日期、启用日期、状态
- 可以按状态、分类、部门、责任人进行查询
- 每条资产至少有一条建账留痕

### 批次 2：固定资产使用闭环

**目标**

让“资产在册之后怎么被人使用”形成闭环，先把领用、调拨、退还做透。

**建议实现范围**

- 建立领用单
- 建立调拨单
- 建立退还单
- 建立交接主单与交接明细，支持一单多资产批量操作
- 每次动作自动写入交接记录
- 回写资产当前使用部门、责任人、位置、状态
- 增加在册、使用中、闲置中的状态转换控制

**控制规则**

- 未正式建账的资产不能领用
- 待处置、已处置、已注销、整改中的资产不能领用
- 领用、调拨、退还不能直接改主档，必须经过单据动作
- 同一交接单中资产类型必须一致，第一期仅允许固定资产
- 批量交接默认采用“全成全败”事务策略，任一资产校验失败则整单回滚
- 调拨与退还结束后必须保留前后责任信息

**建议后端落点**

- 在 `RuoYi-Vue/ruoyi-asset/` 内新增使用单据、服务、校验规则
- 在 `RuoYi-Vue/ruoyi-admin/src/main/java/com/ruoyi/web/controller/asset/` 内增加领用、调拨、退还接口

**建议前端落点**

- 新增 `art-design-pro/src/views/asset/use/`
- 新增 `art-design-pro/src/api/asset/use.ts`

**验收标准**

- 资产管理员可以发起领用、调拨、退还
- 资产管理员可以在单据中批量选择多项固定资产执行同类动作
- 每次动作能改写台账当前归属并保留交接痕迹
- 非法状态下无法进入使用流程

### 批次 3：盘点、处置、财务导出

**目标**

把第一期真正闭环补齐，让资产可以从“使用中”走到“盘点修正”或“待处置/已处置”。

**建议实现范围**

- 建立盘点任务和盘点明细
- 记录盘点正常、盘亏、盘盈、毁损等结果
- 支持盘点结果回写台账或发起处置
- 建立处置登记单
- 支持处置结果确认并回写终态
- 提供财务基础导出
- 提供待盘点、闲置、待处置等基础统计与预警

**控制规则**

- 盘点差异必须选择后续动作
- 资产不能物理删除，只能业务退出
- 待处置资产不能重新进入正常使用
- 处置完成前不能进入终态

**建议后端落点**

- 在 `RuoYi-Vue/ruoyi-asset/` 内新增盘点、处置、导出相关服务
- 在 `RuoYi-Vue/ruoyi-admin/src/main/java/com/ruoyi/web/controller/asset/` 内新增盘点、处置、统计、导出接口

**建议前端落点**

- 新增 `art-design-pro/src/views/asset/inventory/`
- 新增 `art-design-pro/src/views/asset/disposal/`
- 新增 `art-design-pro/src/views/asset/dashboard/`
- 新增 `art-design-pro/src/api/asset/inventory.ts`
- 新增 `art-design-pro/src/api/asset/disposal.ts`

**验收标准**

- 可以发起盘点任务并完成结果登记
- 盘点差异可以修正台账或进入处置
- 财务可以导出第一期需要的基础字段
- 首页或资产看板可以查看总量、状态、待盘点、闲置、待处置

### 批次 4：初始化数据、权限、联调与试运行

**目标**

把“能开发”变成“能试运行”，补齐菜单、角色、字典、初始化数据和上线前测试。

**建议实现范围**

- 初始化菜单、按钮权限、角色权限建议
- 初始化资产分类、状态、处置原因、盘点结果等字典
- 准备一批示例资产数据
- 完成主流程联调
- 完成试运行问题修复
- 输出操作手册和试运行验收清单

**验收标准**

- 资产管理员、财务、查看角色都能进入各自页面
- 第一期开通流程可从头到尾完整走通
- 试运行数据可支持基础查询、导出和问题追踪

## 6. 不建议把下面内容混入第一期

- 不动产占用管理
- 不动产权属登记完整流程
- 不动产巡检整改闭环
- 合同与租赁
- 审批流配置
- 移动端盘点
- 附件档案中心
- 对接采购、合同、财务系统

这些内容都应该作为下一批次或第二期主题，不应与固定资产首个闭环一起推进。

## 7. 当前仓库下的推荐实施顺序

结合仓库现状，推荐顺序不是“先把所有表建完”，而是：

1. 先在后端加 `ruoyi-asset` 业务模块骨架，避免把资产域继续塞进 `ruoyi-system`。
2. 再打通台账主档、分类、状态、基础页面，让资产主档成为唯一底座。
3. 然后接领用、调拨、退还，把交接留痕打通。
4. 最后补盘点、处置、财务导出和统计看板，形成最小闭环。

## 8. 第一批之后的自然扩展方向

当第一期试运行稳定后，再进入下一批次更合理：

- 不动产基础台账和权属基础登记
- 不动产使用分配与占用管理
- 不动产巡检和整改
- 更完整的处置审批
- 附件证照能力
- 与财务或采购系统对接

## 9. 给执行阶段的直接建议

- 如果只允许做一个最小上线版本，请只做到“批次 0 + 批次 1 + 批次 2 + 批次 3”。
- 如果时间更紧，可以把“批次 3”里的预警和看板后移，只保留盘点、处置和导出。
- 如果业务一定要求第一期覆盖不动产，请把不动产范围压缩为“台账登记 + 查询”，不要同时做占用、巡检、整改和处置闭环。

## 10. 前端体验设计补充

资产模块后续页面实现，不应继续按“说明卡片层层叠加”的展示式布局推进，而应统一按“工作台式列表页、强操作任务页、流程表单页、详情查阅页”四种页面分型落地。

详细规范见：

- `docs/plans/2026-03-19-asset-frontend-ux-guidelines.md`

后续 `asset` 目录新增页面或重构页面时，应优先遵循该文档中的骨架、首屏空间、筛选栏位置、批量操作区和固定底部操作条等规范。
