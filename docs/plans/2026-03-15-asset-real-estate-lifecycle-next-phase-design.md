# 资产模块不动产生命周期下一阶段设计

## 背景

截至 2026-03-15，资产模块已经完成以下基线：

- 资产主档、财务、不动产、动态属性、附件、折旧日志的聚合建模与第一轮前后端对接
- 资产分类模板管理、台账实例填值边界、固定资产与不动产生命周期入口分流
- 固定资产的领用、归还、维修、报废/处置与最小审批闭环联调

当前最明显的缺口已经不在固定资产主链路，而在不动产生命周期仍停留在“入口占位、正式动作缺失”的状态。现有前端仅保留“权属变更”“注销/处置”的占位动作，后端也没有对应的正式单据、API、审批回写和验收口径。这会导致系统虽然已经具备统一资产聚合模型，但还没有真正形成“固定资产一条线、不动产一条线”的双分支生命周期结构。

## 设计目标

- 补齐不动产专属生命周期的 MVP，使不动产从“仅能维护台账”提升为“能发起正式业务动作”
- 保持资产主档、财务、不动产专表、动态属性、附件这套聚合台账基线不变，不回退到单表大杂烩
- 延续已经跑通的最小审批闭环，只让关键不动产动作接入审批，不把本阶段重新做成工作流能力扩张项目
- 为本阶段提供最小交付护栏，确保新增动作可以被菜单、权限、契约文档、测试和联调口径完整承接

## 推进方案选择

### 方案 A：不动产生命周期主线优先

- 直接实现权属变更、用途变更、状态变更、注销/处置四类动作
- 收口工作只做局部契约说明和最少测试

优点：

- 业务推进最快
- 与“动作层按资产类型分叉”的既有设计一致

缺点：

- 功能能做出来，但验收、菜单和权限容易继续留尾巴
- 下一轮仍可能需要补一份发布收口计划

### 方案 B：业务主线 + 交付收口双轨推进（推荐）

- 主线：补齐不动产专属生命周期 MVP
- 并行收口线：只补这条主线必须依赖的契约文档、菜单权限、测试矩阵、联调清单与执行说明

优点：

- 既延续既有业务设计，也避免“功能做完但无法稳定验收”
- 不会把注意力重新拉回固定资产遗留问题

缺点：

- 计划拆分要更严格，必须始终坚持“业务主线优先、收口线服务主线”

### 方案 C：发布收口优先，不动产顺延

- 先处理构建阻塞、回归清单、菜单权限、文档补齐，再推迟不动产动作

优点：

- 交付风险最低

缺点：

- 业务价值延后
- 与当前最明显缺口不匹配

## 设计结论

采用方案 B，按“一条业务主线 + 一条并行收口线”推进。

- 业务主线：不动产专属生命周期 MVP
- 并行收口线：本阶段所需的接口契约、菜单权限、测试矩阵、联调清单、执行说明

这意味着下一阶段不是继续重复固定资产收尾，也不是做一次全仓治理，而是在已有稳定基线上，把不动产动作层正式立起来。

## 一、范围与非目标

### 1. 业务主线范围

本阶段只覆盖四类不动产正式动作：

- 权属变更
- 用途变更
- 状态变更
- 注销/处置

其中：

- 权属变更、注销/处置属于关键动作，必须接入最小审批闭环
- 用途变更、状态变更本阶段按“免审批但强留痕”实现，先确保业务动作可用，再保留后续切审批的扩展位

### 2. 并行收口线范围

并行收口线只服务于上述四类动作，包含：

- 后端 API 契约说明
- 前端菜单路径和权限标识
- SQL 执行说明
- 后端服务测试、SQL 契约测试、前端 API/helper 测试
- 主链路联调与验收口径

### 3. 本阶段非目标

- 不切换到 Flowable 或其他重量级工作流引擎
- 不重做资产列表和编辑抽屉整体结构
- 不处理 `monitor` / `system` 目录的历史 TypeScript 构建问题
- 不把固定资产生命周期重新做一轮重构
- 不引入楼栋、房间、分户层级关系等不动产深度建模

## 二、领域分层与边界

### 1. 共享层保持不变

以下能力继续保持共享，不因不动产动作扩展而拆散：

- `asset_info` 作为统一资产主档
- `asset_finance` 作为财务分表
- `asset_real_estate` 作为不动产当前事实表
- 动态属性、附件、折旧日志等聚合子表
- `assetId` 作为统一内部主键

结论：

- 台账层继续维护“当前事实”
- 生命周期层单独维护“动作过程”

### 2. 不动产动作层独立建模

不动产动作层不复用固定资产的 `requisition`、`maintenance`、`disposal` 语义，而是建立独立动作域。建议新增以下四张业务表：

- `asset_real_estate_ownership_change`
- `asset_real_estate_usage_change`
- `asset_real_estate_status_change`
- `asset_real_estate_disposal`

四类动作共享最小引用字段：

- `change_no` 或 `disposal_no`
- `asset_id`
- `asset_no`
- `asset_type`
- `status`
- `apply_user_id`
- `apply_dept_id`
- `reason`
- `effective_date`
- `attachment_json` 或附件引用
- `remark`

其中：

- 权属变更保存前后权利人、产权证号、登记日期、权利类型等关键字段
- 用途变更保存前后土地用途、建筑用途、生效日期
- 状态变更保存前后业务状态、变更原因、生效日期
- 注销/处置保存处置方式、处置原因、终止日期、结果说明

### 3. 主档与动作单职责分离

主资产与不动产专表只保存当前最新事实，不保存完整过程留痕；变更过程统一由动作单承载。

这意味着：

- 查询资产详情时，看见的是当前生效值
- 查询动作台账时，看见的是“前值 -> 后值 -> 原因 -> 附件 -> 审批结果”的过程记录
- 审批拒绝不会污染主资产当前事实

## 三、审批策略与状态流转

### 1. 审批接入策略

本阶段采用“关键动作进审批，普通动作先直达”的策略：

- 权属变更：必须审批
- 注销/处置：必须审批
- 用途变更：免审批，直接完成
- 状态变更：免审批，直接完成

原因：

- 权属变更、注销/处置具有更强的责任与合规属性
- 用途变更、状态变更更适合作为先行 MVP，先打通留痕与主档回写，再评估是否需要切审批

### 2. 状态分层

建议分成两层状态：

- 动作单状态：`draft` / `pending` / `approved` / `rejected` / `completed` / `cancelled`
- 资产事实状态：继续保留在 `asset_info` 与 `asset_real_estate` 中，只表达当前真实状态

映射规则：

- 审批型动作创建后进入 `pending`
- 审批通过后：
  - 权属变更更新 `asset_real_estate` 中的权利相关字段
  - 注销/处置更新资产主档终态和不动产相关结果字段
- 审批驳回后，动作单进入 `rejected`，主档不变
- 免审批动作提交后直接进入 `completed`，并立即回写主档

### 3. 工作流接入边界

审批仍复用现有最小工作流能力：

- `IApprovalEngine`
- `WorkflowTaskController`
- `AssetWorkflowBusinessHandler`

但 `AssetWorkflowBusinessHandler` 需要扩展新的 `businessType`：

- `asset_real_estate_ownership_change`
- `asset_real_estate_disposal`

## 四、前后端职责与数据流

### 1. 后端职责

后端负责：

- 校验资产必须为 `assetType=2`
- 校验动作必填字段与状态前置条件
- 创建动作单并记录前值、目标值、原因、附件
- 对审批型动作发起审批
- 对免审批动作直接完成并回写主档
- 对审批结果执行业务回写

### 2. 前端职责

前端继续以统一资产列表作为入口，不另起一套“第二资产系统”。

具体策略：

- 固定资产继续显示领用、归还、维修、报废/处置
- 不动产显示权属变更、用途变更、状态变更、注销/处置
- 资产列表只负责动作入口与资产上下文传递
- 每个不动产动作落一个独立台账页，负责列表、申请、查看

推荐页面路径：

- `asset/real-estate/ownership/index`
- `asset/real-estate/usage/index`
- `asset/real-estate/status/index`
- `asset/real-estate/disposal/index`

### 3. 数据流

主链路建议统一为：

1. 用户从资产列表或详情发起不动产动作
2. 前端带上 `assetId`、`assetNo`、资产摘要与动作字段提交
3. 后端校验 `assetType=2`
4. 后端创建动作单，并写入前值与目标值
5. 审批型动作进入工作流；免审批动作直接完成
6. 通过业务回写器把结果同步到 `asset_info` / `asset_real_estate`
7. 前端动作台账页与资产列表页展示最新状态

## 五、实施路线

### 批次一：建立不动产动作域基础骨架

目标：

- 建表
- 建立四类动作的领域对象、Mapper、Service、Controller 骨架
- 固化 `assetType=2` 约束

完成标志：

- 四类动作可创建、可列表、可查详情
- SQL 契约和基础类路径稳定

### 批次二：打通审批型关键动作

目标：

- 权属变更 -> 审批 -> 主档回写
- 注销/处置 -> 审批 -> 主档终态回写

完成标志：

- 待办/已办中可看见对应不动产单据
- 审批通过/驳回能正确回写动作单状态与主档状态

### 批次三：补齐免审批动作

目标：

- 用途变更 -> 完成 -> 主档回写
- 状态变更 -> 完成 -> 主档回写

完成标志：

- 不动产当前用途、状态能随着动作单更新
- 过程留痕完整可查

### 批次四：完成前端动作入口与台账联动

目标：

- 列表动作入口从占位升级为正式页面联动
- 新增四个不动产动作页及申请抽屉/详情能力

完成标志：

- 不动产列表中不再出现“可点但必失败”的占位动作
- 新增动作台账页能完成创建、查询、状态回显

### 批次五：并行收口线

目标：

- 补齐菜单 SQL、权限标识、契约说明、测试矩阵、联调说明

完成标志：

- 开发、测试、联调都能沿着同一套文档口径执行
- 能明确区分“本次变更问题”和“历史构建阻塞”

## 六、测试与验收

### 1. 后端

- SQL 契约测试：确认建表脚本、菜单脚本、执行说明已补齐
- 服务测试：覆盖资产类型校验、审批发起、主档回写、驳回回滚
- 工作流回写测试：扩展 `AssetWorkflowBusinessHandlerTest`

### 2. 前端

- API 层测试：覆盖新动作的请求路径与请求体
- helper 测试：覆盖生命周期入口分流与按钮展示
- 页面联调：覆盖列表打开、申请提交、状态刷新

### 3. 联调验收

核心验收链路至少覆盖：

- 权属变更提交 -> 待办审批 -> 通过 -> 不动产主档回写
- 注销/处置提交 -> 待办审批 -> 驳回/通过 -> 台账与主档同步
- 用途变更提交 -> 直接完成 -> 不动产用途回写
- 状态变更提交 -> 直接完成 -> 资产状态展示更新

## 七、风险与对策

### 风险 1：动作表命名和菜单路径再次漂移

对策：

- 建表脚本、菜单脚本、契约说明、前端路径统一采用 `real-estate/*` 命名
- 用 SQL 契约测试锁定路径和权限标识

### 风险 2：审批状态和资产事实状态混用

对策：

- 明确动作单负责过程状态
- 主资产和不动产专表只负责当前事实
- 所有回写都收敛到统一业务处理器或专用服务中

### 风险 3：前端又把不动产动作塞回资产编辑抽屉

对策：

- 维持“统一台账入口 + 独立动作页”结构
- 编辑抽屉继续只服务主数据维护

### 风险 4：全仓构建阻塞掩盖资产模块质量

对策：

- 本阶段以聚焦 Vitest、聚焦 Maven 测试和局部验收为主
- 若 `npm run build` 仍被历史目录阻塞，必须在执行记录中明确注明

## 结论

下一阶段最合适的推进方式不是继续重复固定资产收尾，也不是转向全仓治理，而是在已有聚合台账与最小审批闭环基础上，正式补齐不动产专属生命周期 MVP，并同步完成服务于该主线的最小交付收口。系统完成这一阶段后，资产模块将第一次真正形成“共享台账、分叉动作、关键动作接审批”的稳定结构，为后续更细的不动产动作和流程配置能力留下清晰扩展面。
